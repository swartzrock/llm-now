Evidence

Grounding Context

src/args.ts currently enumerates seven invocation forms, then explains selection, input, aliases, output channels, exit codes, API-key names, and options at equal weight. Three printf forms duplicate the existing rule that input comes from exactly one of --input or stdin.

The README already holds the durable reference material being considered for removal: pipe examples, alias lifecycle and storage paths, stdout/stderr guarantees, diagnostics, and all exit codes. The executable help can therefore specialize in forming a valid command and recognizing supported credential variables.

The runtime export remains the correct source of truth for API-key names. The list should be alphabetized from a copied array, keeping the imported constant immutable. External CLI guidance from GNU, clap, argparse, and clig.dev converges on brief invocation documentation and representative examples rather than an exhaustive syntax matrix.

Arbitration result: retain only facts that prevent a likely usage error or enable first-run provider setup. Everything else belongs in the README.

1

Cargo-inspired quick-start card

Confidence 95%Complexity Low

Description: Keep three representative commands—interactive, saved alias, and explicit provider/model—then replace Selection, Input, and Aliases with a compact Rules block. Borrow Cargo’s restrained terminal hierarchy: green section landmarks, bright literal commands and flags, cool lavender metavariables, and neutral explanatory text. Options follows, then one API key environment variables: heading and the complete alphabetized list.

Section heading literal --flag <replaceable-value> Explanation 
Send a prompt to a selected model.

Usage:
  llm-now --input <text>
  llm-now <alias> --input <text>
  llm-now --provider <id> --model <id|default> --input <text>

Rules:
  Input comes from exactly one of --input or stdin.
  Omit selection for interactive choice; otherwise use an alias or provider/model.
  Model "default" is available only for codex-cli and claude-cli.

Options:
  --input <text>       Prompt text
  --alias <name>       Saved provider/model selection
  --provider <id>      Explicit provider
  --model <id>         Explicit model, or default for a supported CLI provider
  -h, --help           Show help
  --version            Show version

API key environment variables:
  ANTHROPIC_API_KEY
  DEEPINFRA_TOKEN
  DEEPSEEK_API_KEY
  GEMINI_API_KEY
  GOOGLE_API_KEY
  GROQ_API_KEY
  MISTRAL_API_KEY
  OPENAI_API_KEY
  OPENROUTER_API_KEY
  XAI_API_KEY

Color behavior: enable ANSI styling only when help is written to a color-capable stdout TTY. Respect NO_COLOR and TERM=dumb; never emit escapes when help is piped or redirected. The plain-text fallback keeps identical words, spacing, and hierarchy.

Basis
direct: The parser has three real selection paths and one independent input source. The README already owns every removed detail. external: Representative examples are a standard compact-help pattern.
Rationale
Color makes landmarks and syntax roles recognizable before reading, while the compact copy remains understandable without it.
Downsides
The long-form --alias spelling appears only in Options. Color also requires stdout-aware capability detection and dedicated no-color tests.
2

Grammar-first reference

Confidence 90%Complexity Low

Description: Replace every Usage example with one composable synopsis: llm-now [<alias> | --alias <name> | --provider <id> --model <id|default>] [--input <text>]. Follow it with the same three rules, Options, and the single alphabetized key list.

Basis
direct: The synopsis maps cleanly to the parser's interactive, alias, and explicit selection union. external: GNU and mature parser libraries use synopsis grammar for compact reference help.
Rationale
It scales best: future options are added once rather than multiplying a Usage matrix.
Downsides
Bracket-and-pipe notation is efficient for experienced CLI users but less copyable and less immediately friendly.
3

Input-and-selection reference

Confidence 86%Complexity Low

Description: Keep a short example-shaped Usage block, then organize all explanatory copy under one Input and selection: heading. State the exact-one-input rule, the three selection forms, interactive fallback, and the default limitation in three compact lines.

Basis
direct: ParsedArguments models input separately from Selection; the current separate Selection, Input, and Aliases headings scatter those two decisions.
Rationale
It teaches the CLI's mental model in plain language while retaining familiar command examples.
Downsides
A combined heading is less skimmable for someone looking only for alias behavior, though Options still exposes --alias.
4

Credential-reference ordering

Confidence 82%Complexity Low

Description: Keep the three representative commands and compact input/selection guidance, but place the standalone alphabetized API-key list directly after provider guidance and before general Options. The list remains un-mapped: it says which variable names are recognized, not which key belongs to a specific provider.

Basis
direct: The product goal is credential discoverability, and the runtime supplies a flat list rather than provider mappings. reasoned: Proximity to provider selection helps setup scanning without implying stronger semantics.
Rationale
It gives credential setup more visual priority while preserving the requested single heading and complete inventory.
Downsides
Options are the conventional final reference block after Usage; interrupting that flow may feel slightly unusual.
5

Two-column key list

Confidence 70%Complexity Low

Description: Apply a compact two-column presentation to the alphabetized, single-heading variable list. Pair it with any of the first four information architectures; no secondary column header or divider is needed.

Basis
direct: The current ten names are short and uniform. reasoned: Two columns cut the list's height roughly in half while preserving the complete runtime-derived inventory.
Rationale
It recovers terminal height without removing setup information.
Downsides
Column widths and reading order must remain obvious across narrow terminals; a single column is more robust and easier to test exactly.
