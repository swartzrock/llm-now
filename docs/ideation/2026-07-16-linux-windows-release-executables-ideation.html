Linux and Windows release executables

Seven grounded directions for turning llm-now’s three already-tested Linux and Windows archives into trustworthy public release assets—without pulling package-manager publication into the same change.

Grounding Context

Codebase Context. llm-now already defines, builds, and target-natively smoke-tests five deterministic standalone ZIPs: macOS x64/ARM64, glibc Linux x64/ARM64, and Windows x64 baseline. Unsigned release-candidate mode already exact-assembles all five archives and one sorted SHA256SUMS.

The public workflow deliberately narrows that matrix to macOS, signs and notarizes only those two binaries, and requires exactly two final archives. A complete isolated Windows Authenticode job and Linux promotion jobs existed before that narrowing; current policy tests now enforce their absence. The strongest move is therefore restoration and hardening of an existing shape, not invention of a second release system.

Homebrew and Chocolatey templates already consume stable target-specific URLs and checksums, but package-manager publication is explicitly deferred. The useful boundary for this increment is direct GitHub Release assets with clear compatibility and verification guidance.

1

Five-target release batch with trust adapters

Recommended Axis: public promotion Confidence: 95% Complexity: Medium
Description
Use the same five-target native build matrix in candidate and public modes. After native smoke, route macOS archives through signing/notarization, Windows through its signer, and Linux through a verification-only promotion step. Every path emits the same target-explicit release-* artifact contract. One platform-blind finalizer downloads that exact set, regenerates checksums, and publishes only when all five are present.
Basis
direct: The target registry, native smoke matrix, all-five candidate assembly, prior Windows/Linux finishing jobs, and exact-set validator already establish every component. Public mode differs mainly by narrowing the matrix and downloading only release-macos-*.
Rationale
This is the smallest coherent release architecture: build once per supported target, finish according to platform trust rules, then apply one existing completeness invariant. It avoids both a Linux-only special case and a second release pipeline.
Downsides
The final publication becomes a hard five-platform quorum, so a Windows signing outage can block a release that would otherwise succeed on macOS and Linux. The workflow policy tests must change from enforcing macOS-only publication to enforcing the new uniform contract.
Decision it surfaces
Should public publication remain atomic across all supported platforms, or may a platform graduate independently?
2

OIDC-managed Windows signing

Axis: Windows trustConfidence: 82%Complexity: Medium
Description
Use Microsoft Artifact Signing with GitHub OIDC so the protected release job receives temporary signing authority rather than decoding an exportable PFX and password. Keep the signer isolated from repository scripts, verify Authenticode after signing, and repack only the verified llm-now.exe.
Basis
external: Microsoft’s Artifact Signing action supports GitHub OIDC and managed certificates. direct: llm-now’s removed Windows job already established the required protected environment, SHA-256 timestamping, verification, cleanup, and secret isolation.
Rationale
Windows trust is the only new public path that needs publisher identity. Managed signing removes long-lived certificate custody from the repository while restoring the trust boundary that was previously designed.
Downsides
Azure onboarding, certificate-profile eligibility, cost, and regional availability need confirmation. Authenticode improves publisher identity but does not instantly eliminate SmartScreen reputation warnings. If managed signing is unavailable, the prior isolated PFX path is the pragmatic fallback.
3

Protected full-graph rehearsal

Axis: verificationConfidence: 90%Complexity: Medium
Description
Add a protected, non-publishing rehearsal that exercises macOS signing/notarization, Windows signing, Linux promotion, exact five-archive assembly, and final verification. It should use the real protected environments and identities but stop before creating a GitHub Release.
Basis
direct: Current publish:false runs prove all five unsigned artifacts but skip every protected finishing path. Recent release fixes involved notarization verification, signed-archive upload, tag fetch authentication, and publication authorization—failures that ordinary candidate mode could not expose.
Rationale
Adding Windows and Linux increases the number of protected-path edges. A rehearsal turns those edges into pre-release evidence instead of making the next public tag the first integrated test.
Downsides
Signing services may charge per operation, notarization is slow, and a non-publishing run still needs careful version/tag semantics. The rehearsal must not weaken the separate explicit publication authorization gate.
4

Explicit Linux compatibility contract

Axis: Linux compatibilityConfidence: 94%Complexity: Low
Description
Publish Linux assets with precise labels: glibc 2.17 or newer, x64 baseline CPU for broad compatibility, Linux ARM64, and no musl/Alpine support in this release. Put those facts beside the download and in installation guidance rather than relying on internal target names.
Basis
direct: The target registry deliberately selects bun-linux-x64-baseline and bun-linux-arm64, and tests explicitly exclude musl. external: Bun documents glibc 2.17+ and exposes musl as a distinct target family.
Rationale
“Linux support” is too broad to be trustworthy. An explicit artifact-level promise prevents Alpine users from receiving a misleading binary and gives future compatibility testing a concrete floor.
Downsides
Documentation alone does not prove the oldest supported glibc environment. If that floor becomes an important support promise, an older-runtime execution gate should follow.
5

First-release commissioning pass

Axis: rolloutConfidence: 93%Complexity: Low
Description
Treat the first cross-platform public release as commissioning: perform the documented full functional pass on Linux x64 and Windows x64, native smoke on Linux ARM64, checksum verification, execution outside the repository, execution from a path containing spaces, and operation with Bun and Node absent. Record the result before switching the steady-state public matrix to five targets.
Basis
direct: docs/manual-testing.md already assigns those exact platform coverage levels and clean-machine conditions.
Rationale
This gives the new public paths a deeper one-time acceptance test without permanently turning every release into a manual project. It is the safer rollout wrapper around the recommended steady-state architecture.
Downsides
It requires access to clean native environments and a named maintainer checkpoint. The evidence can become stale if platform-finishing behavior changes later, so major signing or packaging changes should trigger recommissioning.
6

Artifact attestations for final assets

Axis: provenanceConfidence: 86%Complexity: Low
Description
Generate GitHub artifact attestations for each final, downloadable archive and the checksum manifest after platform finishing. Keep SHA256SUMS as the simple integrity check; use attestations to bind those bytes to the repository, workflow, and validated commit.
Basis
external: GitHub artifact attestations provide build provenance for release binaries. direct: llm-now already binds releases to a validated tag commit and publishes checksums, but repository search found no provenance or attestation layer.
Rationale
Checksums answer “did these bytes change?” Attestations answer “did the authorized llm-now workflow produce these bytes?” That distinction becomes more valuable as three additional platform assets and future package consumers appear.
Downsides
Attestations add workflow permissions and verification concepts users may not yet recognize. They must cover the post-signing archives, not only unsigned build intermediates, or the provenance story will be incomplete.
7

GitHub assets first; package managers later

Axis: package handoffConfidence: 97%Complexity: Low
Description
Limit this release increment to three new GitHub Release ZIPs and platform-specific install/verify guidance. Preserve stable asset names and checksums so the existing Homebrew and Chocolatey renderers remain valid downstream consumers, but do not add package-manager credentials, repositories, review queues, or publication jobs yet.
Basis
direct: The repository explicitly defers Homebrew and Chocolatey integration while retaining tested templates that consume immutable release URLs and SHA-256 values.
Rationale
This separates the user value—downloadable standalone executables—from a second operational problem: maintaining external registries. Stable direct assets are the prerequisite for package-manager work, not a reason to bundle it into the same milestone.
Downsides
Installation is more manual, and users do not get automatic updates. The README and release notes must therefore make direct download, verification, extraction, and PATH setup easy.
