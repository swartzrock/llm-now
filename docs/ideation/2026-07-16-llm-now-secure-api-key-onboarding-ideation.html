Secure connections for llm-now

A ranked product direction and settled MVP contract for accepting API keys safely, preferring existing environment credentials, storing one fallback key per cloud provider in the native OS vault, preserving alias schema v1, and keeping byok-runtime storage-neutral.

What the evidence supports

Grounding Context

Recommended product direction

Make secure storage additive to today's behavior. For the MVP, each cloud provider may have one native-vault key; a nonempty recognized environment variable remains authoritative, and the vault is used only when that variable is absent. Aliases remain version-1 {provider, model} records with no secret material.

Current codebase

Aliases are strict version-1 {provider, model} JSON records. Persistence is already atomic and permission-hardened for nonsecret metadata. Every cloud call is reconstructed from environment credentials, and diagnostics redact only recognized environment values.

Existing runtime boundary

byok-runtime already accepts either explicit apiKey input or an env credential. Its API deliberately assigns storage, encryption, migration, and UI state to host apps. This feature can begin without changing that contract.

Platform opportunity

Experimental Bun.secrets maps to macOS Keychain, Linux Secret Service, and Windows Credential Manager. It is the simplest native path for the compiled CLI, but its maturity and real behavior must be proven on all five release targets.

Platform limit

Linux Secret Service is a user-session service, not a universal server primitive. SSH, containers, WSL, services, and locked sessions need environment or helper-backed credentials. Storage availability must be treated as a capability, not inferred from process.platform.

Sequence correction: provider before key

Do not ask for an arbitrary key and identify its provider by trial. Key formats overlap and change, and testing one secret against multiple vendors is unnecessary exposure. The safe cloud sequence is provider → credential source → provider-specific validation/model list → model → alias.

Planning continuation · settled scope

Secret Storage MVP Contract

One credential per provider, environment first

The first release stores at most one API key for each cloud provider. Resolution is deterministic: use the provider's nonempty environment variable when present; otherwise read its native-vault record. Credential profiles, multiple accounts per provider, alias v2, and helper-backed storage are deferred.

Key decisions

Credential identityOne saved credential per cloud provider. session-settled: user-directed — chosen over multiple credential profiles for the MVP.Keeps alias schema v1 and avoids account-profile migration before demand proves it necessary.

Product requirements

IDRequirement
R1.For a cloud provider with a nonempty recognized environment variable, llm-now must use that value without reading or prompting for the native-vault value.
R2.When the recognized environment variable is absent or empty, llm-now must resolve the provider's single native-vault credential.
R3.A selected source must fail closed: rejection, revocation, or runtime failure must not cascade to another credential source.
R4.Missing credentials and unavailable, locked, or interaction-required vaults must produce distinct actionable outcomes wherever the exact platform result can be identified reliably.
R5.Direct API-key entry in the MVP must use hidden interactive terminal input; raw keys must not be accepted through arguments, ordinary generation stdin, alias files, or other config files.
R6.Enrollment and replacement must validate the candidate with a provider-specific authenticated operation before attempting to store it.
R7.Aliases must remain version-1 provider/model records; they inherit the provider-level environment-first resolution rule and contain no credential reference.
R8.Every resolved secret must join the diagnostic redaction scope before model listing or generation can expose provider-controlled error text.
R9.The interactive management surface must support add, validated replacement, and deletion for the provider's single vault credential.
R10.Native-vault onboarding must remain disabled on any release target until the pinned Bun version's compiled executable passes real get, set, replace, and delete checks in a representative user session.

Delivery sequence and gates

0 · Compatibility gate

Pin the exact Bun API shape and test compiled get/set/replace/delete behavior on macOS x64/arm64, Linux x64/arm64 with an active Secret Service session, and Windows x64. Record platform errors before defining normalized vault outcomes.

1 · Resolver and redaction

Add the host-owned vault boundary, environment-first resolution, explicit runtime-key injection, fail-closed source selection, and secret-aware diagnostic sanitization. Keep alias persistence unchanged.

2 · Enrollment and lifecycle

Add provider-first hidden entry, authenticated validation, save or replace consent, deletion, cancellation safety, and clear handling for missing or unavailable vault services.

3 · Release confidence

Exercise source precedence and negative paths in automated tests, run native-vault checks in representative user sessions, and document environment-only operation wherever secure storage is unavailable.

Scope boundaries

NowDeferredNever automatic
One vault credential per cloud provider; environment-first resolution; hidden interactive enrollment; validated replacement; deletion; alias v1; resolved-secret redaction. Multiple accounts, named credential profiles, alias v2 binding, external helper protocol, scripted auth stdin, credential sync, and a broad doctor/lifecycle suite. Raw-key arguments, secrets in alias/config JSON, plaintext storage, self-encrypted local files, cross-provider key inference, or fallback to a second source after authentication failure.

Known limitation accepted for the MVP

Because aliases remain provider/model records, they do not pin account identity. A process-level environment key can intentionally override the saved provider key. Named profiles and alias-bound identity should return only when real multi-account demand justifies schema v2.

Recommended starting point

The verified connection ceremony

An interactive zero-argument invocation becomes a setup lane, not promptless generation. It shows existing aliases and local/CLI providers first; “Add API key” then performs provider → credential source or hidden key → provider-specific model-list validation → model → optional alias → commit. Setup exits with a copyable next command rather than sending a model request with no prompt.

Axis
Credential intake and consent
Basis
direct: Prompt resolution currently happens before selection in src/app.ts, so a bare TTY invocation cannot onboard. byok-runtime already uses provider-specific model listing as authenticated verification. Existing alias capture proves optional, cancellation-safe persistence.
Rationale
This is the shortest trustworthy path from fresh install to llm-now gpt56 --input "…". Provider-first avoids unreliable key inference and prevents a key from being tested against multiple vendors.
Downsides
Changes zero-argument semantics and requires a real setup state machine plus masked input. It must remain isolated from scripted generation so existing stdin and exit-code contracts do not drift.

Adoption without surprise

Environment keys graduate only by consent

When discovery finds OPENAI_API_KEY or another recognized variable, llm-now uses it immediately under existing semantics. After provider-specific validation, an interactive user can explicitly copy it into the native store as a fallback. The prompt shows provider, environment-variable name, destination, and a masked fingerprint—never the value—and defaults to No. It does not run in CI/headless contexts and never removes or edits the environment variable.

Axis
Credential intake and consent
Basis
direct: Current discovery observes environment-backed providers without mutating configuration; alias overwrite already uses contextual default-No consent. external: GitHub CLI separates environment-token use from explicit secure-store login.
Rationale
Existing users get a one-confirmation migration path while ephemeral, employer-injected, or CI credentials remain ephemeral by default.
Downsides
Promotion creates a second copy that can later drift. Environment-first precedence means the saved copy remains dormant until the variable disappears, so the consent prompt must describe it as a fallback rather than an active replacement.

Deferred identity model

Credential profiles after MVP

If real multi-account demand appears, introduce stable, nonsecret profiles such as openai/work and openai/personal. A future alias schema can reference a profile while the secret remains in its backend. The MVP instead stores one vault credential per provider and preserves alias schema v1.

Axis
Alias and credential identity
Basis
direct: Alias v1 currently permits exactly provider/model, tests prohibit credential fields, and equality cannot distinguish two accounts with the same provider/model. reasoned: Credential identity must be independent of command identity for safe sharing and rotation.
Rationale
Profiles remain the clean path to deterministic account identity without placing keys in JSON, but they should be earned by multi-account usage rather than imposed on first-release storage.
Downsides
Deferral accepts provider-level ambient identity: an environment key can override the stored key for every alias using that provider. A later v2 migration still needs explicit legacy behavior.
Confidence
91%
Complexity
Medium

Deferred structure: if multi-account demand justifies alias v2, aliases express execution intent, credential profiles own account identity, and secret backends own values.

Cross-platform security

Native-vault first, secure-only fallback policy

Use a small host-owned credential-store adapter with Bun.secrets as the preferred implementation, but only after compiled-artifact tests prove get/set/delete across the supported targets and relevant session contexts. Runtime outcomes distinguish usable storage from locked/unavailable/interaction-required failures. Automatic selection may choose native storage or a configured helper; it never silently writes a plaintext or self-encrypted file.

Axis
Secure-store portability
Basis
external: Bun maps its experimental secrets API to native stores on all three OS families. Secret Service documents session, lock, and prompt behavior. direct: llm-now already compiles and smokes five native targets but has no secret-store coverage.
Rationale
The architecture stays simple on desktop without making a false promise to headless Linux or WSL. “Secure storage” becomes a verified product capability, not a platform-name branch.
Downsides
The Bun API is experimental and does not provide a formal structured probe contract; llm-now must normalize observed operation failures carefully. Real keychain tests may need dedicated CI runners or manual release gates.

Secret-bearing channels

Stage secret lanes by audience

The MVP gives humans hidden terminal enrollment and keeps existing environment variables as the only scripted credential lane; generation stdin remains exclusively prompt text. A later release can add a dedicated auth-stdin command and shell-free helper for enterprise systems. Raw keys are never accepted as positional values or flags.

Axis
Headless automation and recovery
Basis
direct: Current stdin is consumed as prompt text and noninteractive execution requires deterministic selection. external: GitHub CLI accepts tokens on stdin; Docker and Anthropic establish helper-based runtime secret resolution; CWE-214 documents argv exposure.
Rationale
This answers “should users pass a key to llm-now?” precisely: interactively through a dedicated hidden-input ceremony, or externally through an environment variable—not through generation syntax. Helper extensibility can follow after native storage is proven.
Downsides
The MVP does not provide a pipe-friendly command that copies a secret into the vault. Scripted users must continue managing environment credentials outside llm-now.

88%
Complexity
Medium–High

Illustrative split: credential input has dedicated channels; generation keeps its current prompt contract.

Failure philosophy

Stable source, explicit repair

Source selection is fixed before a provider call: environment when present, otherwise the provider's one vault credential. Missing, locked, revoked, rejected, or deleted credentials fail with the most reliable reason and remediation the platform exposes; a rejected environment key never cascades to the vault. Interactive repair may validate and replace or delete the stored provider credential.

Axis
Headless automation and recovery
Basis
direct: Existing stale/missing/corrupt aliases fail without discovery or substitution, and runtime errors preserve stage/provider context. Alias metadata writes already use collision-aware locking and preserve prior files on failure.
Rationale
Fail-closed source selection avoids “helpfully” switching accounts behind a script while preserving the current authority of process-level environment configuration.
Downsides
Alias v1 cannot pin account identity, so changing the environment changes the account used by every alias for that provider. The MVP accepts this compatibility tradeoff.

Library recommendation

Host-owned credential resolution; byok-runtime stays clean

The first product increment needs no byok-runtime storage change. llm-now can retrieve a secret, pass the existing explicit { apiKey } config for model listing/generation, compose its own available connections, and expand diagnostic redaction to include resolved values. Keep Keychain, Secret Service, Credential Manager, config, aliases, and user prompts entirely out of the runtime package.

If a second host later needs the same discovery behavior, add only a storage-neutral input such as availableCredentialProviders or hasCredential(provider). That lets canonical provider discovery include host-backed availability without reading a secret or adding keyring dependencies.

Axis
Runtime/library boundary
Basis
direct: byok-runtime already accepts explicit API keys, centralizes provider/env metadata, and explicitly assigns persistence to host apps. Its public-contract test deliberately excludes storage/app dependencies. Env-only discovery is the one real seam a future availability hint would address.
Rationale
This prevents a product-specific keychain implementation from hardening into a general library burden. llm-now can validate the product model first; the runtime abstraction appears only after reuse is real.
Downsides
llm-now initially owns more connection-composition logic. If that logic duplicates provider ordering or metadata, drift is possible; keep it narrow and use already-exported provider/env metadata.
Confidence
95%
Complexity
Low initially
