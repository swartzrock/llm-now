Evidence before ideas

Grounding Context

Codebase context. package.json is already the single version source for CLI output, archive filenames, validators, and packaging. The current public workflow is manually dispatched with a pre-existing tag and then builds five native targets, signs and notarizes macOS, assembles the exact asset set, verifies checksums, issues attestations, and creates a GitHub Release behind protected environments.

Reference pattern. byok-runtime uses @changesets/cli and changesets/action to maintain a chore: release PR. Its publication step targets npm, so llm-now should copy the version-PR half, not changeset publish.

External constraints. Changesets officially separates versioning from publication and supports versioning private packages. A workflow should not rely on a tag pushed with GITHUB_TOKEN to wake a second workflow; an explicit reusable-workflow call is the reliable handoff.

Institutional context. This repository has no docs/solutions/ corpus or CONCEPTS.md, so the current workflows, release policy tests, maintainer docs, and sibling repository are the governing evidence.

IDEA 1 · TOP PICK

Reviewed release train with protected promotion

Description: Feature PRs add Changesets intent. A low-privilege push-to-main coordinator creates or refreshes one reviewable release PR that consumes those files, bumps package.json, and writes CHANGELOG.md. Merging that generated PR explicitly invokes the existing binary pipeline for its exact merge SHA; final signing and publication remain protected.

Conceptual flow only; implementation boundaries belong in the downstream requirements and plan.

Axis · Release-PR orchestration Confidence · 94% Complexity · High
Basis
direct: byok-runtime proves the release-PR loop; llm-now already propagates one validated SHA through its five-platform workflow. external: an explicit call avoids GitHub token recursion limits.
Rationale
This gives the user the requested “changeset triggers a bumped release” behavior without equating a feature merge with immediate public publication.
Downsides
Requires reshaping the existing manual workflow into a coordinator plus reusable release engine and commissioning bot-authored release PR behavior.
IDEA 2

Changesets as a version-only intent compiler

Description: Add the same authoring surface as byok-runtime—bun run changeset, .changeset/config.json, a contributor README, and a generated changelog—but configure private-package versioning and omit npm publication. Multiple pending Changesets coalesce into the next version PR, with the highest applicable SemVer bump.

Axis · Contributor/version intent Confidence · 93% Complexity · Low
Basis
external: official Changesets lifecycle separates version from publish and supports private-package versioning. direct: llm-now's package.json version already drives every user-visible binary version surface.
Rationale
The useful part of Changesets is semantic intent and a reviewed version/changelog diff; the npm publisher is incidental to byok-runtime and unnecessary here.
Downsides
Contributors must learn when to add patch, minor, or major intent. Whether non-release PRs need an explicit declaration remains a policy choice.
IDEA 3

One reusable binary engine, two entry points

Description: Extract the current build/sign/assemble/attest/publish graph behind a reusable workflow interface keyed by version and source SHA. The Changesets coordinator calls it for normal releases, while workflow_dispatch remains a clearly labeled recovery and unsigned-candidate entrance to the same engine.

Axis · Release-PR orchestration Confidence · 90% Complexity · High
Basis
direct: the current workflow already has a validated SHA handoff and separate candidate/public paths. reasoned: one engine prevents the automatic and emergency paths from drifting.
Rationale
The normal path becomes simple without deleting a proven operator escape hatch for commissioning, runner failures, or exceptional recovery.
Downsides
Reusable-workflow permissions, secrets, environment approvals, and caller inputs need careful validation; this is the largest structural change.
IDEA 4

Tag-last publication with exact-SHA resumability

Description: Build, sign, notarize, assemble, and checksum every final asset before creating vX.Y.Z. In the protected publication job, create the tag if absent; if it exists, continue only when its peeled commit equals the exact release SHA. A mismatched tag fails closed, and an existing GitHub Release is never overwritten.

Axis · Protected binary publication Confidence · 88% Complexity · Medium
Basis
direct: today's workflow creates no release until final checks pass but requires the tag before building; it already revalidates remote tag SHA and refuses duplicate releases.
Rationale
A public version identity should mean releasable bytes exist. Exact-SHA resume semantics preserve retry safety if failure occurs between tag and Release creation.
Downsides
Tag creation moves into a privileged transaction. The team must define and test every partial state around tag push, attestation, and Release creation.
IDEA 5

Desired-version reconciliation state machine

Description: Treat the merged package.json version as desired state and the remote tag/GitHub Release as observed state. No tag or release starts work; a matching completed release is a no-op; an exact-SHA tag without a release is resumable; conflicting or impossible partial states fail with a precise operator message.

Axis · Retry/idempotency Confidence · 85% Complexity · Medium
Basis
external: Changesets warns that “no pending changesets” also occurs on unrelated post-release pushes. direct: llm-now already refuses duplicate releases and checks exact tag/SHA identity.
Rationale
Automation should reconcile toward the intended release rather than depend on one perfect event. That makes runner, signing, and approval failures recoverable without empty commits.
Downsides
A state table adds policy surface and must avoid accidentally normalizing dangerous partial states; only a small set should be resumable.
IDEA 6

Least-privilege permission airlock

Description: Give the Changesets job only the permissions needed to update the version PR. Keep native builds read-only, Apple credentials only in release-signing, and contents/OIDC/attestation writes only in release-publication. The coordinator passes version and SHA across that boundary, never secrets.

Axis · Protected binary publication Confidence · 92% Complexity · Medium
Basis
direct: llm-now's release policy tests already enforce narrow publication permissions and protected environments; byok-runtime's simpler npm workflow uses broader workflow-level writes.
Rationale
Copying byok-runtime's ergonomics should not widen the trusted surface of a signed, attested binary release.
Downsides
GitHub reusable-workflow permission inheritance is subtle. The final design must prove that the caller cannot grant or consume authority outside the intended jobs.
IDEA 7

Two independent concurrency lanes

Description: Serialize and cancel superseded release-PR refreshes on main, but key binary publication to the settled version/SHA and never cancel an active signing or publication run. New feature merges may update the next release queue without interrupting the release already in promotion.

Axis · Retry/idempotency Confidence · 82% Complexity · Low
Basis
direct: CI already uses cancel-in-progress concurrency, the current release workflow has no concurrency, and byok-runtime serializes its main-branch release workflow.
Rationale
Version-PR freshness and public promotion have opposite cancellation semantics. Separating them prevents ordinary development activity from superseding protected release work.
Downsides
Concurrency keys must distinguish versions while preventing duplicate work for the same version; careless grouping can block a legitimate recovery run.
What did not survive

Rejection Summary

#IdeaReason Rejected
1Changeset-or-no-release declaration on every PRUseful policy variant, but not necessary for first adoption and may add contributor friction.
2Release PR as executable manifest including lockfileBasis refuted: Bun's root lockfile does not encode the package version.
3Add a new release-PR native preflightExisting PR CI already builds and smokes all five targets; only bot-PR commissioning remains.
4Generate a separate machine-readable release contractGrounded, but added schema and plumbing are premature while version + SHA + target registry already carry the core identity.
5Release dashboardChangesets PR metadata plus GitHub checks and artifacts already provide the surface; a second dashboard adds little value.
6Read-only classifierUnderspecified and subsumed by the explicit reconciliation state machine and permission airlock.
7Sealed asset consignment across runsNo durable immutable cross-run store, retention model, or lookup mechanism was justified.
8Preview and production share one immutable manifestA pre-merge preview cannot contain the eventual merge SHA; it needs a finalization boundary and adds complexity.
9Immediate release for every individual changesetFails the cost/cadence test: each release consumes a five-platform matrix and protected signing stages; batching is stronger.
10Copy byok-runtime's changeset publishScope mismatch: llm-now is a private binary CLI with GitHub assets, not an npm publication target.
11Create the tag before the build and rely on tag eventsFragile failure semantics and GitHub token recursion make this weaker than explicit same-run orchestration.
12Remove manual dispatch entirelyEliminates a valuable commissioning and recovery entrance without reducing the underlying release engine.
Recommendation

Choose Idea 1 as the architecture. Treat Ideas 2, 3, 4, and 6 as required parts of its first implementation. Add Ideas 5 and 7 as explicit operational contracts during brainstorming so retry and concurrency behavior cannot remain accidental.
