Decision at a glance

Goal Capsule

Objective

Let a user with any supported LLM already available install llm-now and receive generated text within 60 seconds of launching it.

Product authority

llm-now owns terminal discovery, selection, aliases, diagnostics, and native distribution. byok-runtime owns provider adapters and runtime operations.

Open blockers

No product decisions block planning. Planning must verify compiled-runtime compatibility, signing credentials, and the publication path for Homebrew and Chocolatey.

Product behavior and boundaries

Product Contract

Summary

llm-now will be a separate open-source project that ships self-contained native executables for mainstream macOS, Linux, and Windows systems. It discovers providers through byok-runtime, guides an interactive first call, and supports deterministic one-shot reuse through global aliases.

Problem Frame

Calling a local or already-authenticated LLM from a shell often requires installing plugins, writing provider configuration, or understanding provider-specific endpoints before receiving any output. The user's concrete AIChat setup required an OpenAI-compatible client entry, a fake key, and manual Ollama configuration; the official llm quick start requires an Ollama plugin.

The desired advantage is not a larger feature set. It is the shortest path from “an LLM is already available here” to a successful response, without provider configuration files or credential management.

Key Decisions

Separate product and repository.

The CLI has its own identity, release cadence, package-manager metadata, and user contract while depending on the public runtime library.

Native-only distribution.

Permanent installations through Homebrew and Chocolatey replace the earlier npx/bunx entry points.

One-shot, not chat.

The product generates one response from --input or stdin and stores no conversation history.

Runtime-owned providers.

The CLI adds no provider adapters or plugin system; new providers arrive through byok-runtime.

Interactive humans, deterministic scripts.

TTY users can choose from menus; non-interactive calls must identify an alias or provider and model.

Aliases contain no secrets.

A user-global alias stores only a provider/model pair and relies on environment keys or existing CLI authentication at execution time.

Actors

A1.First-time evaluator

Has at least one supported local server, authenticated AI CLI, or cloud key and wants a successful response without configuration.

A2.Recurring shell user

Uses a saved alias for repeatable one-shot prompts and shell pipelines.

A3.Automation caller

Runs without a TTY and needs deterministic selection, clean stdout, and meaningful failure exits.

A4.Release maintainer

Publishes one version across the supported binary matrix and package-manager channels.

Key Flows

F1.First interactive success

Trigger: A TTY user runs llm-now --input "Reply with a short one-line poem". The CLI discovers candidates, presents a provider menu, lists that provider's models, presents a model menu, generates text, prints the response, and then offers to save the working pair as an alias.

F2.No provider available

Trigger: Discovery returns no candidates. The CLI reports what it checked and gives the shortest manual setup path for supported local servers, AI CLIs, or environment-backed cloud providers; it installs nothing.

F3.Saved alias call

Trigger: A user runs llm-now --alias llamqwen --input "Explain recursion". The CLI resolves the global provider/model pair, generates once, and prints only the response to stdout.

F4.Non-interactive call

Trigger: stdin is piped or no TTY is present. The CLI requires an alias or explicit provider/model selection and fails with actionable diagnostics rather than silently choosing.

F5.Native release

Trigger: A version is released. Automation builds and tests the mainstream platform matrix, publishes checksummed GitHub Release artifacts, and updates the Homebrew and Chocolatey packages that install them.

Requirements

Product boundary

IDRequirement
R1.llm-now must be maintained as a separate repository and product from byok-runtime.
R2.The CLI must use byok-runtime as the only source of provider adapters, model listing, and text generation.
R3.The CLI must support one-shot text generation from --input and stdin without storing conversations.
R4.The CLI must not store API keys or authentication tokens.

Discovery and selection

IDRequirement
R5.The CLI must discover all provider candidates returned by byok-runtime without requiring provider configuration files.
R6.An interactive first call without a saved selection must present a provider menu followed by a model menu.
R7.The CLI must generate only after the user selects a provider and model or supplies a deterministic selection.
R8.A non-interactive call must require --alias or explicit provider and model inputs.
R9.When no providers are found, the CLI must report checked provider classes and actionable manual setup guidance.
R10.Discovery must not install software, start provider runtimes, pull models, or create credentials.

Aliases and user configuration

IDRequirement
R11.After successful generation, an interactive call must offer to save the working provider/model pair under a user-chosen alias.
R12.A saved alias must contain only provider and model identifiers.
R13.User aliases must be global to the current operating-system account and available from every working directory.
R14.macOS and Linux configuration must honor $XDG_CONFIG_HOME and fall back to ~/.config/llm-now; planning must select the equivalent Windows user-config location.

Terminal and automation contract

IDRequirement
R15.Successful generation must write response text only to stdout.
R16.Menus, selection context, and operational diagnostics must not contaminate response stdout.
R17.Failures must return nonzero exit codes and identify the stage that failed without exposing credential values.
R18.Help output must document interactive, alias, explicit-selection, stdin, configuration, and diagnostic behavior.

Native distribution

IDRequirement
R19.Release artifacts must be self-contained executables that do not require Node.js or Bun on the user's machine.
R20.The initial release matrix must cover macOS x64, macOS ARM64, Linux x64, Linux ARM64, and Windows x64.
R21.Every released binary must pass the same user-facing command and behavioral contract.
R22.GitHub Actions must build and test the binary matrix and publish versioned, checksummed GitHub Release artifacts.
R23.Homebrew must provide supported installation and upgrade paths on macOS and Linux.
R24.Chocolatey must provide supported installation and upgrade paths on Windows.
R25.The release process must verify installed package-manager binaries, not only raw build artifacts.
R26.bun build --compile is the preferred packaging mechanism, while self-contained compatibility remains the governing requirement.

Acceptance Examples

AE1.First poem

Covers R5–R7, R11, R15–R16. With Ollama and an authenticated Claude CLI available, a TTY call presents both providers, lists models only after the provider choice, prints the poem alone to stdout, and offers to save the successful pair.

AE2.No providers

Covers R9–R10, R17. With no local endpoint, supported executable, or recognized key present, the call exits nonzero and shows manual next steps without modifying the machine.

AE3.Alias reuse

Covers R11–R14. After a successful call saves llamqwen, the alias works from a different directory and resolves the same provider/model without storing credentials.

AE4.Broken alias target

Covers R12, R17. If the saved provider or model is no longer usable, the call fails clearly and does not silently select a replacement.

AE5.Pipeline safety

Covers R8, R15–R17. A piped call without deterministic selection fails before generation; a piped call with an alias emits only model response text to stdout.

AE6.Cross-platform installation

Covers R19–R25. A released version installs through Homebrew on a supported macOS/Linux target and Chocolatey on Windows x64, runs without Node.js or Bun installed, and reports the same version and help contract.

Success Criteria

  • A user with at least one supported provider already available receives generated text within 60 seconds of launching the installed CLI.
  • The first successful call requires no provider configuration file, plugin installation, credential copy, or endpoint entry.
  • The same one-shot and alias workflows pass on every target in the initial native matrix.
  • Homebrew and Chocolatey installations are exercised as release gates.
  • No alias or diagnostic artifact contains credential values.

Scope Boundaries

Deferred for later

  • Windows ARM64.
  • Linux musl artifacts.
  • Baseline and modern x64 variants.
  • Additional package managers or installer scripts.

Outside this product's identity

  • Interactive chat, sessions, or history.
  • RAG, attachments, tools, or agents.
  • Credential storage or key setup.
  • Provider plugins or CLI-owned adapters.
  • Installing runtimes or downloading models.
  • npx, bunx, or npm-distributed execution.

Dependencies / Assumptions

  • The public Node surface of @swartzrock/byok-runtime remains the runtime dependency after PR #22 is merged and released.
  • The CLI owns configuration assembly, selection policy, terminal formatting, and exit behavior even though provider adapters remain in the library.
  • Bun standalone executables remain compatible with the runtime's HTTP and child-process behavior across the target matrix.
  • GitHub Releases provide stable URLs and checksums suitable for Homebrew and Chocolatey packaging.
  • Release maintainers can obtain any signing credentials required for a smooth macOS and Windows experience.

Build prompt for llm-now
Build a new open-source project named `llm-now` in its own repository.

Product promise:
“One prompt. Any available LLM. Now.”

Primary outcome:
A user who already has any supported LLM provider available should install the native CLI, launch it, and receive generated text within 60 seconds without writing provider configuration, installing plugins, copying credentials, or entering endpoints.

Before coding:
1. Verify that swartzrock/byok-runtime PR #22 has merged and that the published `@swartzrock/byok-runtime` version exposes the required public Node APIs.
2. Inspect the current public exports and documentation. Use public package entrypoints only.
3. Verify `bun build --compile` compatibility with every required target, especially HTTP calls, PATH-based executable discovery, and spawned Codex/Claude commands.
4. Produce a concise implementation plan and identify any true blockers. Do not ask questions whose answers are already specified below.

Product boundary:
- `llm-now` owns terminal UX, discovery orchestration, provider/model selection, user-global aliases, diagnostics, stdout/stderr behavior, native builds, and release packaging.
- `@swartzrock/byok-runtime` owns provider adapters, provider discovery primitives, model listing, text generation, credentials, and local process execution.
- Do not duplicate provider adapters in this repository.

Required user flows:

1. Interactive first call
   `llm-now --input "Reply with a short one-line poem"`
   - Discover every provider candidate exposed by `byok-runtime`.
   - Present a provider menu.
   - List models for the chosen provider and present a model menu.
   - Generate one response.
   - Print response text only to stdout.
   - After success, offer to save the provider/model pair under a user-chosen alias.

2. Alias reuse
   `llm-now --alias llamqwen --input "Explain recursion"`
   - Resolve the saved provider/model pair.
   - Never store credentials in the alias.
   - Do not silently replace a broken or unavailable alias target.

3. Stdin automation
   - Accept prompt text from stdin.
   - When no TTY is available, require `--alias` or explicit `--provider` and `--model`.
   - Never silently select a provider or model for a non-interactive call.
   - Keep response stdout clean; send diagnostics elsewhere.

4. No-provider diagnosis
   - Report which provider classes were checked.
   - Give the shortest manual next steps for supported local servers, authenticated AI CLIs, or environment-backed cloud providers.
   - Do not install software, start runtimes, pull models, or create/store credentials.

Configuration contract:
- Aliases are global to the current OS user and contain only provider and model identifiers.
- On macOS and Linux, honor `$XDG_CONFIG_HOME` and fall back to `~/.config/llm-now`.
- Select and document the platform-appropriate Windows user-config location.
- Configuration writes must be safe against partial/corrupt updates.

Scope constraints:
- One-shot text generation only.
- No interactive chat, conversation history, RAG, attachments, tools, agents, credential manager, plugin system, or CLI-owned provider adapters.
- No npm, `npx`, or `bunx` distribution path.
- Provider support must expand only through `byok-runtime`.

Native distribution requirements:
- Produce self-contained executables that require neither Node.js nor Bun at runtime.
- Initial targets:
  - macOS x64
  - macOS ARM64
  - Linux x64
  - Linux ARM64
  - Windows x64
- Prefer `bun build --compile`; the requirement is working standalone binaries, so surface any incompatibility rather than hiding it.
- Use GitHub Actions to build and test the full matrix.
- Publish versioned, checksummed GitHub Release artifacts.
- Support installation and upgrades through Homebrew on macOS/Linux and Chocolatey on Windows.
- Exercise the installed Homebrew and Chocolatey packages as release validation, not only raw binaries.
- Determine and implement required signing/notarization steps.

Quality and security requirements:
- Response text only on stdout after successful generation.
- Menus and diagnostics must not contaminate response stdout.
- Failures return nonzero exit codes and identify the failed stage.
- Never display, log, persist, or include credential values in diagnostics.
- Help must document interactive, alias, explicit provider/model, stdin, configuration, and diagnostic behavior.
- Test every conditional flow above, including cancellation, no providers, model-list failure, generation failure, stale aliases, non-interactive misuse, and cross-platform path/config behavior.

Success criteria:
- From launching an installed CLI, a user with one available supported provider gets text in under 60 seconds.
- No provider configuration file is needed for the first successful call.
- The same behavioral contract passes on all five target binaries.
- Homebrew and Chocolatey installation paths are release-tested.
- Alias/configuration files contain no secrets.

Working process:
- Use a focused feature branch and pull request workflow.
- Keep implementation phases small and verifiable.
- Add automated tests before or with each behavior.
- Use official Bun, GitHub Actions, Homebrew, Chocolatey, and `byok-runtime` documentation as primary sources.
- Do not publish public releases or package-manager entries without confirming credentials and release authorization with me.

Start by reporting:
1. the verified `byok-runtime` API surface and minimum compatible version;
2. the proposed implementation phases;
3. risks in native compilation, process spawning, signing, Homebrew, and Chocolatey;
4. the exact automated verification strategy.

Then proceed through implementation unless a true blocker requires my decision.
