What the evidence says

Grounding Context

Codebase Context. @swartzrock/byok-runtime is currently a library-only ESM package for Node 20+. It publishes . and ./node, has no bin, and excludes examples from the tarball. The smoke CLI imports repository-private source paths, so it cannot be promoted unchanged.

The reusable Node surface is already emerging: findAvailableProviders detects local endpoints, installed Codex/Claude commands, and configured API-key environment variables. But those signals are not equivalent—reachable, installed, configured, authenticated, and generation-ready are different states. The current ByokProviderId[] result loses that distinction.

Externally, npm and Bun make a package binary the standard zero-install contract. Diagnostic tools often ship separately from their SDKs, while same-package binaries work when the executable and library are one cohesive lifecycle. Mature general LLM CLIs also show the scope trap: general generation quickly pulls in key storage, history, plugins, configuration, and model management.

Decision signal: command-line access is worth offering, but the differentiated product is environment readiness and proof—not another interactive AI shell.

Six directions survived

Ranked Ideas

  1. Public readiness API + companion doctor
  2. Structured readiness report
  3. Inspect → list → prove ladder
  4. Secret-free automation receipts
  5. Optional armed run --auto
  6. Same-package thin binary
1.

Two-surface product: public readiness API + companion doctor CLI

Confidence 90%Complexity Medium
Description
Publish a focused companion such as @swartzrock/byok-runtime-cli or @swartzrock/byok-runtime-doctor, runnable through npx and bunx. Its identity is diagnostic: show what the host can use, where readiness stops, and optionally prove a selected provider. It consumes only public Node-library APIs.
Axis
Distribution and package boundary
Basis
direct: the core package publishes only dist and two library exports, while the current CLI is private and source-relative. external: AI SDK DevTools and MCP Inspector use companion diagnostic packages rather than adding operational tooling to their core SDK surfaces.
Rationale
This gives users the useful zero-install path without coupling every library installation and release to a growing command interface. The package boundary also communicates that environment inspection and local process execution are explicit tooling choices.
Downsides
A second package adds naming, version coordination, release automation, and discoverability work. For only three tiny commands it may initially feel heavier than a same-package binary.
Top pick. It best preserves the runtime's app-agnostic identity while giving the CLI enough room to become a genuinely good diagnostics product.
2.

Structured provider readiness report in the Node library

Confidence 92%Complexity Medium
Description
Add a Node-library capability that returns structured evidence for every provider: discovery signal, readiness state, safe failure category, and available next check. TypeScript hosts, Electron settings screens, CI, and the CLI would all consume the same report.
Axis
Shared core and maintenance
Basis
direct: findAvailableProviders already performs three distinct checks but reduces them to ByokProviderId[]. The example then reconstructs configuration and command mappings outside the published boundary.
Rationale
The CLI question reveals a missing reusable abstraction. Moving evidence—not argument parsing—into the library prevents semantic drift and makes readiness useful even for callers that never install the executable.
Downsides
Readiness states become a public contract that must remain stable as providers evolve. The API must avoid implying authentication or generation success when only ambient configuration was detected.
3.

A visible readiness ladder: inspect → list → prove

Confidence 91%Complexity Medium
Description
Design the command surface around escalating effects. inspect is passive and secret-safe; models may authenticate or execute a local CLI; prove performs a minimal generation and may spend tokens. Human and JSON output both identify the level reached.
Axis
Command and product scope
Basis
direct: provider discovery currently checks endpoints, executable presence, and env-var presence without launching installed AI CLIs. Model listing and generation cross materially different authentication, process-execution, and cost boundaries.
Rationale
The ladder turns the vague word “available” into a useful diagnosis and makes consent visible before consequential actions. It is a stronger product model than exposing library method names one-for-one.
Downsides
More states and verbs require careful naming. Some providers blur the ladder because model listing itself may be unavailable or as consequential as a minimal request.
4.

Secret-free receipts and a machine-first I/O contract

Confidence 88%Complexity Medium
Description
Make every diagnostic operation available as stable JSON, with provider evidence, readiness state, timestamps, runtime version, and sanitized failure classes. Preserve stdout for data, stderr for diagnostics, stdin for prompts, and documented exit categories; report credential variable names/presence, never values.
Axis
Automation and security contract
Basis
direct: the current CLI already keeps values on stdout and errors on stderr, but collapses all failures to exit code 1 and discards why a provider was discovered. The runtime promises not to persist or log credential values.
Rationale
A secret-free receipt becomes useful in CI, desktop setup screens, installers, support bundles, and bug reports. That cross-surface reuse is more durable than terminal prose.
Downsides
JSON schemas and exit semantics create compatibility commitments. Detailed failures must be carefully sanitized across third-party API and CLI output.
5.

Optional, explicitly armed run --auto smoke path

Confidence 74%Complexity Medium
Description
Keep one-shot generation as a proof primitive, not the CLI's identity. An explicit command discovers candidates, lists models, attempts generation with documented fallback, emits selected provider/model and failures on stderr or JSON, and leaves response text alone on stdout.
Axis
Command and product scope
Basis
direct: examples/first-available-llm.sh and its tests already demonstrate discovery, model selection, fallback, and response-only stdout. reasoned: generation is the only definitive proof of readiness, but does not require interactive chat, history, or key storage.
Rationale
This provides the memorable “use whatever already works” demo while keeping the diagnostics-first boundary intact. It also replaces brittle shell orchestration with one supported operation.
Downsides
Automatic provider/model policy may conflict with the project's host-owned-policy stance; random selection is unsuitable as a default public contract. It may incur cost or execute local authenticated tools, so consent and deterministic policy are essential.
6.

Same-package thin binary as the simpler alternative

Confidence 68%Complexity Low–Medium
Description
Add a bin to @swartzrock/byok-runtime so the obvious package name works for imports, npx, and bunx. Keep the executable dependency-light, semver-aligned, and limited to parsing/formatting over public Node APIs.
Axis
Distribution and package boundary
Basis
external: npm and Bun directly support package-declared binaries; Playwright demonstrates that a same-package library and executable can be coherent when they share one lifecycle. direct: the runtime already has a Node-only public boundary suitable for the executable.
Rationale
This is the shortest route to “library or command line” and avoids a second package name. It is attractive if the CLI remains permanently small and releases should always move in lockstep.
Downsides
Every library consumer receives the binary and its support promise. CLI UX changes force core releases, and a future expansion would make separation harder after users depend on the package name.
Decision rule: choose this only if the permanent CLI promise fits on one screen of help and remains a projection of the core runtime—not an independently growing product.
